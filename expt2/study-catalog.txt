STUDY DESIGN
------------
Goal: Quantify how much fused-MoE latency depends on GPU P2P communication
(NVSwitch/NVLink) vs. PCIe fallback, by intentionally degrading NCCL transport.

Shape: Qwen3-30B-A3B (hidden=2048, inter=768, E=128, topk=8, bf16)
Backend: allgather_reducescatter
Routing: uniform (fixed for isolation)
Tokens: [1, 2, 4, 8, 16, 32, 64, 128, 512, 2048, 8192]
Warmup: 30 iters | Measure: 100 iters | Seed: 42

Transport conditions:
  nvlink_default     : no env vars (NVLink/NVSwitch baseline)
  no_nvls_no_p2p     : NCCL_NVLS_ENABLE=0 + NCCL_P2P_DISABLE=1 (PCIe fallback)
  no_nvls_no_p2p_1ch : above + NCCL_MAX_NCHANNELS=1 (single-channel PCIe)

Parallel points:
  tp1-ep1  : control (no inter-GPU collective — identical across all transport conditions)
  tp1-ep2  : onset of transport sensitivity
  tp1-ep4  : max EP on 4-GPU node
  tp2-ep2  : TP allreduce + EP allgather both active

Total: 3 transport × 4 parallel points = 12 SLURM jobs, 132 benchmark conditions

Hardware target: PACE ICE H200 SXM5 (atl1-1-03-017-30-0)
  Note: H100 nodes all have bad GPUs (see KNOWN ISSUES below). H200 has same
  NVSwitch architecture so results are equivalent for this study.

Config: expt2/configs/study.transport-conditions.qwen3.yaml
Cluster output: /storage/ice1/0/2/sghose7/moe-breakdown-runs/expt2/transport-conditions-qwen3/
SLURM logs: ~/scratch/moe-breakdown-runs/expt2/transport-conditions-qwen3/slurm-logs/

================================================================================
ROOT CAUSE: NCCL CUMEM + SLURM --gpus-per-task INCOMPATIBILITY (RESOLVED)
================================================================================

Problem diagnosed 2026-06-27 via NCCL_DEBUG=INFO on H200 (atl1-1-03-017-30-0).

Root cause:
  NCCL 2.21+ defaults to P2P/CUMEM (CUDA Unified Memory) for intra-node P2P.
  SLURM --gpus-per-task=1 sets CUDA_VISIBLE_DEVICES=0 per rank, hiding peer GPUs.
  When NCCL calls ncclP2pImportShareableBuffer it fails with:
    Cuda failure 101 'invalid device ordinal'
  because the peer GPU's physical busId (e.g. 0x5d000) is not visible in the
  process's CUDA device list. This cascades to "NCCL error: unhandled cuda error"
  in PyNcclCommunicator.all_reduce() during vllm's warmup call.

Confirmation: torch.distributed NCCL allReduce succeeded on the same nodes because
  those test runs did NOT use --gpus-per-task=1 (all GPUs visible to all ranks).

This is NOT a cluster hardware issue. NVLink topology is intact (NV18 confirmed).
This is a NCCL 2.30 / SLURM gpu-per-task isolation incompatibility.

Fix applied in expt2/slurm/run_direct_moe_sweep.sbatch:
  export NCCL_CUMEM_ENABLE="${NCCL_CUMEM_ENABLE:-0}"
  Forces legacy IPC P2P transport instead of CUMEM. Peer GPUs do not need to be
  in CUDA_VISIBLE_DEVICES for IPC handles.

NCCL debug findings (for reference):
  - MNNVL fabric UUID 0.0 cliqueId 0x0 state 3 healthMask 0x0 (fabricmanager not
    driving MNNVL, but irrelevant — P2P/CUMEM is intra-node, not NVLink SHARP)
  - P2P/CUMEM selected for all 24 channels before CUMEM import fails
  - Both H100 and H200 nodes affected (same NCCL version, same SLURM config)
