STUDY DESIGN
------------
Goal: Quantify how much fused-MoE latency depends on GPU P2P communication
(NVSwitch/NVLink) vs. PCIe fallback, by intentionally degrading NCCL transport.

Shape: Qwen3-30B-A3B (hidden=2048, inter=768, E=128, topk=8, bf16)
Backend: allgather_reducescatter
Routing: uniform (fixed for isolation)
Tokens: [1, 2, 4, 8, 16, 32, 64, 128, 512, 2048, 8192]
Warmup: 30 iters | Measure: 100 iters | Seed: 42

Transport conditions:
  nvlink_default     : no env vars (NVLink/NVSwitch baseline)
  no_nvls_no_p2p     : NCCL_NVLS_ENABLE=0 + NCCL_P2P_DISABLE=1 (PCIe fallback)
  no_nvls_no_p2p_1ch : above + NCCL_MAX_NCHANNELS=1 (single-channel PCIe)

Parallel points:
  tp1-ep1  : control (no inter-GPU collective — identical across all transport conditions)
  tp1-ep2  : onset of transport sensitivity
  tp1-ep4  : max EP on 4-GPU node
  tp2-ep2  : TP allreduce + EP allgather both active

Total: 3 transport × 4 parallel points = 12 SLURM jobs, 132 benchmark conditions

ENGINEERING NOTES
================================================================================

NCCL P2P / SLURM --gpus-per-task incompatibility (resolved 2026-06-27):
  NCCL 2.30 selects P2P/CUMEM (or P2P/IPC) for intra-node transfers.
  Both require peer GPUs in CUDA_VISIBLE_DEVICES. SLURM --gpus-per-task=1
  restricts each rank to CUDA_VISIBLE_DEVICES=0 only, so peer import fails
  with "invalid device ordinal" / "invalid argument" → "unhandled cuda error".
  Fix: removed --gpus-per-task=1 from srun. All job-allocated GPUs stay visible;
  each rank calls torch.cuda.set_device(local_rank) in distributed.py.
  Confirmed working: P2P/IPC channels established, both ranks return OK.

Custom allreduce patch:
  vllm's CustomAllreduce calls can_device_access_peer(rank, world_size) using
  global rank as device ID, which breaks when only device 0 is visible. Patched
  to always return False in distributed.py (forces NCCL allreduce instead).
