OVERVIEW
--------
Compares deepep_low_latency and deepep_high_throughput all2all backends against
each other (and implicitly against the NCCL allgather_reducescatter baseline
from expt2/expt2.5) on the same Qwen3-30B-A3B shape and H200 hardware.

Research thesis (full study suite):
  Network and bandwidth are now first-class bottlenecks for MoE LLM inference —
  more so than for dense models — and this effect is amplified by routing imbalance.
  We show this through:
    1. Backend comparison: LL vs HT at different token counts (decode vs prefill)
    2. Routing sweep: how expert load imbalance interacts with each backend
    3. Bandwidth congestion: routing × transport (NVLink vs PCIe fallback)
    4. Resource budget: time breakdown into network vs compute (bucket profiling)

Backend summary:
  deepep_low_latency    : one-sided RDMA dispatch via NVSHMEM; optimized for
                          decode-scale token counts; lower per-call overhead but
                          less bandwidth-efficient at large batches.
                          On single-node H200: uses NVSHMEM NVLink P2P transport.
                          NOTE: Empirically slower than HT at all token counts on
                          H200 — likely because NVSHMEM P2P is slower than NCCL
                          NVSwitch collectives. LL is designed for IB RDMA
                          (multi-node). Single-node comparison favors HT.
  deepep_high_throughput: collective-based dispatch using NCCL; better bandwidth
                          utilization at prefill-scale; affected by NCCL transport
                          conditions (NVLink vs PCIe fallback).

At ep=1: no inter-GPU expert dispatch → both backends are identical.
         Use tp1-ep1 as a sanity check / pure compute baseline.
At ep>1: expert tokens route across GPUs; LL via NVSHMEM, HT via NCCL all2all.

Shape: Qwen3-30B-A3B (hidden=2048, inter=768, E=128, topk=8, bf16)
Hardware: PACE ICE H200 SXM5, single-node, 4 GPUs/node
Warmup: 30 iters | Measure: 100 iters | Seed: 42

ANALYSIS GUIDE — TIME BUDGET DECOMPOSITION
================================================================================

The bucket profiler records per-iteration CUDA events for:
  bucket_max_rank_network_ms  : time in NCCL collectives (allgather + reducescatter)
  bucket_max_rank_gpu_compute_ms : time in GPU compute (expert GEMMs)

Derived metrics:
  network_fraction = network_ms / (network_ms + gpu_compute_ms)
  achieved_bw_GBps = allgather_recv_bytes / (network_ms * 1e-3 * 1e9)

Caveat for DeepEP LL: network_ms = 0 for LL runs. NVSHMEM dispatch ops are not
captured by NCCL CUDA events. Total latency (latency_median_ms_max_rank) remains
valid; the decomposition is unavailable for LL. This is a known instrumentation gap.
  → To show time budget for LL, future work: add CUDA event wrappers around the
    NVSHMEM dispatch calls in moe_kernel.py.

For DeepEP HT: NCCL events DO capture the collective dispatch, so network_ms/
gpu_compute_ms breakdown is valid. Use Study E to show how routing skew shifts the
network fraction under bandwidth congestion.

Key comparisons enabled by the full study suite:
  1. Study A vs B (same conditions): LL latency vs HT latency across token counts
     → Shows LL is slower on single-node H200 (NVSHMEM P2P < NCCL NVSwitch)
  2. Study C: Does routing skew increase LL latency? And does it scale with skew?
     → LL queue depth hypothesis: heavy skew overflows per-expert dispatch queues
  3. Study D vs Study B: Does adding routing skew to HT matter at all token counts?
     → HT all-to-all is symmetric, but data volume asymmetry may still hurt
  4. Study E (nvlink vs no_nvls_no_p2p): Does routing skew amplify bandwidth sensitivity?
     → Key claim: skewed routing under PCIe fallback is disproportionately slow
  5. Across Studies B/D/E vs expt2.5 routing sweep: Does DeepEP HT help or hurt
     under bandwidth congestion compared to plain NCCL allgather_reducescatter?

ENGINEERING NOTES
================================================================================

DeepEP LL mode token limit (discovered on first ep>1 run, 2026-06-29):
  The DeepEP legacy buffer sizes the combine receive buffer as:
    num_experts * max_tokens * msg_size * 4 buffers / 16 < INT_MAX
  With hidden=2048, num_experts=128:
    msg_size = 4160 bytes
    Limit: max_tokens < 16384 (128 * 16384 * 4160 * 4 / 16 = 2,181,038,080 > INT_MAX)
    Safe max: 8192 (ratio = 1,090,519,040 < INT_MAX)
  Studies A/C token range capped accordingly. HT mode is unaffected.

NVSHMEM IB transport fix (revised 2026-06-29):
  NVSHMEM 3.3.24 has no env var to disable transport plugins — it loads all
  .so files from its lib dir via dlopen and tries to init each. On PACE ICE,
  IBRC/IBDEVX/IBGDA crash with SIGSEGV during init (IB verb permissions not
  granted to user-space jobs). NVSHMEM_DISABLE_IBRC / NVSHMEM_TRANSPORT are
  NOT recognized env vars in NVSHMEM 3.3.24 (silently ignored).
  LD_LIBRARY_PATH prepend approach failed: libnvshmem_host.so has DT_RPATH=${ORIGIN},
  which takes priority over LD_LIBRARY_PATH.
  Fix: replace the actual transport .so files in the venv at their RPATH location
  with stubs (nvshmemt_init returning 1 = failure). Originals backed up as .so.3.orig.
  Replaced: /storage/ice1/0/2/sghose7/moe-breakdown-deepep-venv/lib/
    python3.11/site-packages/nvidia/nvshmem/lib/
      nvshmem_transport_ibrc.so.3   → stub
      nvshmem_transport_ibdevx.so.3 → stub
      nvshmem_transport_ibgda.so.3  → stub
  Set in slurm/run_direct_moe_sweep.sbatch:
    LD_LIBRARY_PATH=/home/hice1/sghose7/nvshmem-stubs:$LD_LIBRARY_PATH (belt+suspenders)
    NVSHMEM_BOOTSTRAP=uid (avoids PMI/MPI dependency)

NVSHMEM_QP_DEPTH assertion:
  deepep low_latency_dispatch asserts:
    nvshmem_qp_depth >= (num_max_dispatch_tokens_per_rank + 1) * 2
  With max_tokens=8192: need (8192+1)*2 = 16386.
  NVSHMEM_QP_DEPTH=16384 fails by 2. Fix: NVSHMEM_QP_DEPTH=32768 (set in sbatch).

DeepEP tp>1 limitation (LL only):
  vLLM's DeepEP LL modular_kernel.py:807 asserts a1.size(0)==num_experts/ep.
  LL prepare path returns a1 with num_experts/(tp*ep) rows (re-sharded by TP).
  For tp=2, ep=2: 128/(2*2)=32 rows but assertion expects 128/2=64. FAILS.
  HT kernel path does not have this assertion. tp2-ep2 only valid for HT.

Prior NVSHMEM blocker (expt1, H100):
  expt1 limited DeepEP to tp1-ep1 only due to NVSHMEM IBRC failures.
  With the IBRC/IBGDA disable fix and H200 NVLink, ep>1 now works on H200.
  First confirmed ep>1 DeepEP run on this cluster: jobs 5449022 (tp1-ep2),
  5449023 (tp1-ep4).
